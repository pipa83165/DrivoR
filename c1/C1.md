# C1 — 冻结 VGGT-Ω 并联几何分支(增益上界探针 / Phase 1 试金石)

## 目的与假设
不计推理成本地把教师几何信息**以最短路径**送进规划 decoder,测出几何先验的增益上界(H1 的直接检验)。C1 是全项目的"定价"实验:B 系的一切恢复率都以 C1 − A1 为分母。**定位:探针,不是部署方案**(端到端延迟预计 ~250ms,vs DrivoR 110ms)。

## 精确配置(含超参)

### 教师前向
- 冻结 VGGT-Ω 1B(默认 512 版;Phase 0 可改判 256-Text-Alignment 版),eval() + no_grad + bf16,**不加载 camera/depth head**
- **4 相机联合前向**:前视 = reference frame,相机顺序固定 **前 / 前左 / 前右 / 后**(写入指纹)
- 教师预处理管线与 DrivoR 主干**完全解耦**,且**必须与官方 `vggt_omega.utils.load_fn.load_and_preprocess_images` 保持一致(直接 import,禁止手写复刻)**:
  - 默认 `mode="balanced"`, `image_resolution=512`(= 面积 ≈512²、AR 保持,官方默认与训练分布一致;NAVSIM ~16:9 输入约得 672×384 级别,尺寸自动为 patch16 倍数)
  - `mode="max_size"`(长边 512,~512×336)**不作默认**,降级为 C3a 效率变体
  - ⚠️ 交接稿原写"长边 ~512(如 336×448)"对应 max_size 约定,与官方默认冲突——已按官方 balanced 修正(2026-07-04)
  - **教师输入不做颜色增广**;归一化统计量以 load_fn 内部实现为准(不手抄常量)
  - register 切片按官方约定:`camera_and_register_tokens[:, :, :1]` = camera token(默认丢弃),`[:, :, 1:]` = 16 registers
- 取每相机 16 个 register → **64 个几何 token**,维度 D(名义 2048,语义待核实——见问题 3)

### 融合
- `geo_proj`:线性投影 D→256 + 分支 embedding + 相机 embedding
- 与 DrivoR 原生 64 scene tokens 拼接 → **128×256 联合 memory**,进 trajectory 与 scoring 两个 decoder 的 cross-attention
- **融合点严格限于 decoder memory,不碰主干 ViT**(这是"并联"的定义;任何流向主干的路径都改变实验语义)
- 高范数 register 前置 LN(否则 attention sink)

### 训练
- 同 A1 协议:navtrain 10 epochs,AdamW lr 2e-4 cosine,batch 16,4×A100,损失全 1,只动 perception 侧(geo_proj + register/LoRA + decoders)
- 3 seeds

## 与基线的差异点
相对 A1 仅新增:教师 token 缓存读取 → geo_proj → memory 拼接。主干、decoder 结构、损失、数据全部不变。

## 内部对照(必跑,精确定义)

| 对照 | 精确定义 | 回答的问题 |
|---|---|---|
| **C1-shuffle** | **整样本级**错配:样本 i 的几何 token 换成随机 partner 样本 j≠i 的(整组 64 token 一起换,不做 token 级打乱),partner 在**数据加载层从 split 全体缓存 token 中抽取**,**每次样本访问重新随机配对**;训练与评测**一致错配**(评测也 shuffle),同一 provider 实现,任意 batch size(含 batch=1)可执行。⚠️ 原定义"batch 内错配"在 run_pdm_score batch=1 下不可执行,2026-07-04 修订为 split 级抽样(统计等价:错配内容仍来自同分布真实特征) | 增益是"几何内容"还是"多了 64 个统计量正常的 token"(容量/正则效应) |
| **C1-noise** | 高斯噪声 token,**匹配缓存全体的均值/方差**(per-dim 统计),**注入在投影前的 D=2048 维空间**(不是投影后),再走同一 geo_proj | 比 shuffle 更弱的下界:连"真实特征流形"都没有时 decoder 会不会自己找到用法 |
| **C1-drop** | 训练带几何 token,**推理时截断 memory 回 64**(禁止置零实现——零 key 仍分走 softmax 质量造成稀释伪影,见问题 2) | 训练后模型对几何 token 的**依赖度读数**:drop 后回落幅度 = 依赖度 |

## 机制分析钩子
- decoder cross-attention 质量分配:预期 scoring 给几何 token **25–40%**、trajectory **10–20%**
- 几何 token 相机间余弦矩阵(看后视是否坍缩,与 DrivoR 原生 token 对照)
- 场景切片:动态密集 / 路口
- 深度线性探针(几何 token 上)
- 相机 dropout 鲁棒性曲线

## 已知坑与预案(实现级,除上述 5 问题外)
1. 分支/相机 embedding 记得**不参与**指纹(它们是学生侧可学习参数),但拼接顺序参与。
2. 评测时若某样本缓存缺失:直接 raise,禁止静默回退到 drop 模式(会污染均值)。
3. shuffle 的随机配对种子与训练种子解耦并记录,保证可复现。
4. 训练/评测 dataloader 的 shuffle 对照必须走同一实现路径(同一函数、同一开关),防止训练错配评测不错配的半吊子状态。
